## Install from npm

Install is the one-time setup. Once `maestro` is on your `PATH`, you can start a local loop in any repo and come back to the same control surface when work is in flight.

The published npm package is the fastest path to install or upgrade the Docker-backed launcher:

<DocsCommandField command={`npm install -g @olhapi/maestro`} />

The installed command name is still `maestro`.

This package installs a thin host launcher, not a native platform binary. Docker is required for normal CLI use.

The npm launcher requires Node 24 or newer because it uses the built-in
`node:sqlite` module to inspect Maestro workflows before Docker starts.

The launcher resolves its runtime image in this order:

- `MAESTRO_IMAGE`
- the locally pinned image from `~/.maestro/launcher/runtime.json`
- the npm package version you installed

Pull and pin the latest runtime image explicitly:

<DocsCommandField command={`maestro self-update`} />

Validate Docker access, the runtime pin, and launcher prerequisites:

<DocsCommandField command={`maestro doctor install`} />

## Install the Maestro skill bundle

If you want Codex and Claude Code to pick up Maestro-specific guidance automatically, install the bundled skill once:

<DocsCommandField command={`maestro install --skills`} />

That copies the skill into `~/.agents/skills/maestro` for Codex and `~/.claude/skills/maestro` for Claude Code.

## Install without npm

If you want the same launcher without npm, use the repository install script:

<DocsCommandField command={`curl -fsSL https://raw.githubusercontent.com/olhapi/maestro/main/scripts/install_maestro.sh | sh`} />

## Docker image

Use the published GHCR image when you want a containerized install path:

<DocsCommandField command={`docker pull ghcr.io/olhapi/maestro:latest`} />

The image entrypoint is `maestro`. Its default command is `run --db /data/maestro.db`, so mounting only a data directory is enough to start the shared daemon with the container defaults.

## Build from source

For local development or unsupported platforms, build the binary directly from the repo:

<DocsCommandField command={`go build -o maestro ./cmd/maestro`} />

Contributor Docker build:

<DocsCommandField command={`docker build -t maestro-local .`} />

## What to install first

- Install Docker before you try to run the launcher-installed `maestro` command.
- Install `go` if you need source builds or local development.
- Install `codex` before you attempt real orchestration runs that launch the Codex CLI.
- Install an MCP-capable coding agent if you want to connect to the local queue interactively.
- Use the npm package or the curl installer when you want the shortest path from machine setup to the `mcp add` command or config entry your coding agent expects.

## Contributor tooling

If you are contributing from a repo checkout, run the root install once before anything else:

<DocsCommandField command={`pnpm install`} />

That single install does three things:

- installs the repo-managed Git hooks through Husky
- bootstraps the shared workspace across `apps/frontend` and `apps/website`
- makes the root helper scripts available for common docs-site tasks

If you contribute regularly across multiple machines or rely on CI often, you can also enable Turborepo Remote Cache locally with:

<DocsCommandField command={`pnpm exec turbo login && pnpm exec turbo link`} />

For website work, the most useful root commands are:

- `pnpm verify` for the full local verification suite before pushing
- `pnpm run verify:pre-push` for the exact pre-push hook suite, including real Docker image smoke, tarball install smoke, local-registry install smoke, curl-installer smoke, and retry-safety coverage
- `pnpm run verify:ci` for the lean GitHub Actions gate
- `pnpm run website:dev` for the local Astro docs server
- `pnpm run website:check` for type and content validation
- `pnpm run website:test` for the website test suite
- `pnpm run website:build` for a production build check

The hooks stay targeted. Frontend changes run frontend lint and tests, website changes run Astro checks and website tests, workspace and hook changes run the full `pnpm verify` suite, and `pre-push` runs `pnpm run verify:pre-push` before the push proceeds with Docker/image/distribution smoke and the retry-safety harness included. GitHub Actions stays on the smaller `pnpm run verify:ci` gate.
