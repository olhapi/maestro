## 1. Initialize a workflow file

Start in the repo you want Maestro to orchestrate:

<DocsCommandField command={`maestro workflow init .`} />

This writes a repo-local `WORKFLOW.md` with the default Kanban tracker settings, Codex command, and prompt template.

## 2. Expose the tracker to MCP clients

If you installed Maestro from npm, add the shipped `maestro` command to Codex once:

<DocsCommandField command={`codex mcp add maestro -- maestro mcp`} />

That uses the npm-installed `maestro` binary already on your `PATH`; you do not need to point Codex at a manually built executable.

`maestro mcp` is a stdio bridge into the live `maestro run` daemon for the same database. Save the entry first, then start the daemon and let Codex attach through it.

## 3. Start the orchestrator

Start the daemon:

<DocsCommandField command={`maestro run`} />

If `--db` is omitted, Maestro uses `~/.maestro/maestro.db` by default. If `--port` is omitted, Maestro serves the dashboard and HTTP APIs on `http://127.0.0.1:8787`.

Running `maestro run` without `repo_path` starts the shared daemon for the current database. It does not infer the repo from your shell working directory.

The loop is straightforward:

1. Maestro loads `WORKFLOW.md`.
2. It polls for issues in the `ready` state.
3. It creates per-issue workspaces.
4. It dispatches work to the configured agent command.
5. It tracks retries, logs, runtime status, and the MCP transport used by `maestro mcp`.

## 4. Create a first queue from Codex

With the daemon running, ask Codex to seed the tracker through the saved `maestro` MCP server:

<DocsCommandField
  command={`codex --cd /path/to/todo-app "Use the maestro MCP server to create a project, epics, and issues for a new react todo app. Set the project repo path to the current working directory. Plan the features first, split them into epics, add a priority value and blockers for each issue, set their state to ready."`}
/>

This example assumes the default local `kanban` provider, which supports epics, issue priorities, and blockers.

## 5. Open the control center

By default, `maestro run` serves the embedded dashboard, the live observability API, and the richer dashboard application API on the same HTTP server. Use `--port` to change the port.

- Dashboard: `http://127.0.0.1:8787/`
- Live API root: `http://127.0.0.1:8787/api/v1/`
- Dashboard API root: `http://127.0.0.1:8787/api/v1/app/`

If you want CLI helpers to talk to that live daemon, pass `--api-url`:

<DocsCommandField
  command={`maestro status --dashboard --api-url http://127.0.0.1:8787
maestro sessions --api-url http://127.0.0.1:8787
maestro project start PRJ-1 --api-url http://127.0.0.1:8787
maestro project stop PRJ-1 --api-url http://127.0.0.1:8787`}
/>

The preview warning on `run` is intentional. Pass `--i-understand-that-this-will-be-running-without-the-usual-guardrails` only when unattended Codex execution is actually what you want.
